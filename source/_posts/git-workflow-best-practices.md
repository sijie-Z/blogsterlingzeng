## Git 不只是版本控制

刚学 Git 的时候，我以为它就是"保存文件的高级版本"。`git add`、`git commit`、`git push`，三连搞定。

后来进入团队，才发现 Git 的核心不是"版本控制"，而是**协作**。它解决的问题是：多个人同时修改同一份代码，怎么不互相覆盖、不丢失工作、不引入 bug。

理解了这个，很多 Git 的"最佳实践"就不是死记硬背的规则，而是从协作需求中自然推导出来的。

## 提交信息的重要性

### 我见过最糟糕的提交记录

```
fix
update
bug
修改
asdf
test
```

这种提交记录在单人项目里无所谓，但在团队里是灾难。三个月后你看到一个 bug，想通过 `git log` 找到是谁、为什么改了这行代码，结果看到的全是"fix"和"update"。

### 我现在遵循的规范

```
<type>(<scope>): <description>
```

- `feat`: 新功能
- `fix`: 修复 bug
- `refactor`: 重构（不改变功能）
- `docs`: 文档
- `style`: 格式调整
- `test`: 测试
- `chore`: 构建/工具

```
feat(auth): 添加 OAuth2 登录支持
fix(api): 修复用户列表分页参数错误
refactor(utils): 提取日期格式化逻辑
```

**为什么这样写？**

不是因为"规范说要这样写"，而是因为这样的提交信息能回答三个问题：
1. **做了什么**（type + description）
2. **影响哪里**（scope）
3. **为什么做**（从 description 推断）

当你需要回溯问题、生成变更日志、或者理解代码历史的时候，这三个信息就是救命稻草。

## 分支策略的选择

### Git Flow：适合有明确版本周期的项目

```
main ──────────────────────── 生产环境
  │
develop ───────────────────── 开发主线
  │
  ├── feature/search ──────── 搜索功能
  ├── feature/auth ─────────── 认证功能
  └── feature/profile ──────── 个人资料
```

优点：清晰、严谨、适合大团队
缺点：流程重、分支多、合并冲突多

### GitHub Flow：适合持续部署的项目

```
main ──────────────────────── 生产环境
  │
  └── feature/search ──────── 开发完成 → PR → 合并 → 部署
```

优点：简单、快速、适合小团队
缺点：不适合需要多版本维护的项目

### 我的选择

大多数项目用 **GitHub Flow** 就够了。只有在需要维护多个版本（比如同时维护 v1 和 v2）的时候，才考虑 Git Flow。

**不要为了"看起来专业"而选择复杂的流程。** 三个人的团队用 Git Flow，就像三个人的公司设五个部门一样荒谬。

## 代码审查的真正价值

### 审查不是找茬

我见过有人审查代码的时候，专门找命名不规范、缩进不对这种问题。这些当然重要，但不是代码审查的核心价值。

代码审查的真正价值是：

1. **发现逻辑漏洞** — 作者可能有思维盲区，审查者能从不同角度看问题
2. **知识传递** — 团队成员通过审查了解彼此的代码和思路
3. **统一风格** — 不是靠规范文档，而是靠实际代码的互相学习
4. **提高质量** — 知道有人会看你的代码，你自然会写得更认真

### 我的审查原则

**作为审查者：**
- 先理解意图，再看实现
- 指出问题的同时，给出建议
- 区分"必须改"和"建议改"
- 肯定做得好的地方

**作为被审查者：**
- 不要把审查意见当成人身攻击
- 解释你的设计决策，但保持开放
- 如果不同意，说明理由，不要默默忽略

## 实用技巧

### 交互式 rebase：整理提交历史

```bash
# 整理最近 3 个提交
git rebase -i HEAD~3
```

在 PR 提交之前，用交互式 rebase 把零散的提交整理成有意义的几个。这不是"改写历史"，而是"编辑草稿"——你不会把草稿里的每一笔修改都保留在最终版里。

### 暂存工作：处理紧急任务

```bash
# 当前工作还没完成，但需要切分支处理紧急 bug
git stash
git checkout main
# 修复 bug...
git checkout feature-branch
git stash pop
```

### 二分查找：定位引入 bug 的提交

```bash
# 从最近 100 个提交中找到引入 bug 的那个
git bisect start
git bisect bad          # 当前版本有 bug
git bisect good HEAD~100  # 100 个提交前没有 bug

# Git 会自动二分，让你测试每个中间版本
# 测试后标记 good 或 bad，直到找到罪魁祸首
```

## 我犯过的错误

### 错误 1：在 main 分支上直接开发

"就改一行代码，不用开分支了吧。"

结果那一行代码引入了一个 bug，直接影响了部署。从此以后，哪怕只改一个字符，我也开分支。

### 错误 2：提交信息写"fix"

"反正是修 bug，写 fix 就行了。"

三个月后我想找那个 bug 的修复记录，搜 "fix" 出来 47 条结果。

### 错误 3：不审查就合并

"这个 PR 看起来很简单，直接合并吧。"

结果那个"简单"的 PR 引入了一个隐蔽的类型错误，在生产环境暴露了。

### 错误 4：长期不合并 develop 到 feature 分支

"等我开发完再合并。"

结果开发了两周，合并的时候有 200 多个冲突。从此以后，我每天早上第一件事就是把 develop 合并到我的 feature 分支。

## 总结

Git 工作流没有"最佳实践"，只有"最适合的实践"。但有几个原则是通用的：

1. **提交信息要有意义** — 不是为了规范，是为了未来的自己和队友
2. **分支策略要匹配团队规模** — 不要过度设计
3. **代码审查是学习机会** — 不是找茬，是互相提升
4. **犯错不可怕，重复犯错才可怕** — 从错误中建立习惯

最后分享一句我很喜欢的话：

> 代码是写给人看的，顺便能让机器执行。

Git 的提交历史也是一样——它是项目的叙事，是团队的集体记忆。好好写它。
