## 为什么要有自己的博客

在这个信息爆炸的时代，写博客似乎有点"过时"。公众号、知乎、掘金、Twitter——发内容的平台太多了，为什么要折腾一个自己的博客？

我的理由很简单：**所有权。**

在别人的平台上写东西，你永远是"租客"。平台改规则、改算法、甚至关站，你的内容就没了。我见过太多优秀的技术博客因为平台变动而消失。

自己的博客不一样。域名是你的，代码是你的，内容是你的。哪怕十年后 Hexo 不维护了，你的 Markdown 文件还在，换个框架一样能用。

## 技术选型：我比较了什么

### WordPress

最老牌的 CMS，功能强大，插件丰富。但对我来说太重了——我只需要一个写文章的地方，不需要用户管理、电商插件、SEO 套件。而且 PHP + MySQL 的部署成本比静态站点高得多。

### Hugo

用 Go 写的静态站点生成器，速度极快。但它的模板语言 Go Template 让我很不舒服——语法奇怪，调试困难。社区也没有 Hexo 活跃。

### Jekyll

GitHub Pages 原生支持，部署最简单。但 Ruby 生态我不熟悉，而且构建速度比 Hexo 慢。

### Hexo

最终选了 Hexo，原因：

1. **Node.js 生态** — 我是前端开发者，Node.js 是我的主场
2. **主题丰富** — 社区活跃，主题质量高
3. **Markdown 原生支持** — 纯文本写作，Git 友好
4. **部署灵活** — 可以部署到任何静态托管服务

**但最重要的原因是：Redefine 主题。** 它的设计感、功能丰富度和文档质量都让我满意。一个好的主题能省掉 80% 的定制工作。

## 主题选择：为什么是 Redefine

我试过这些主题：

| 主题 | 优点 | 缺点 |
|------|------|------|
| Butterfly | 功能最多，文档最全 | 设计偏"花哨"，配置项太多 |
| NexT | 经典、稳定 | 设计略显过时 |
| Fluid | 简洁、现代 | 功能相对少 |
| Redefine | 设计感强、功能够用 | 相对较新 |

选择 Redefine 的关键因素：

1. **设计语言** — 毛玻璃效果、圆角卡片、柔和的阴影，现代但不浮夸
2. **单页应用体验** — 基于 swup.js，页面切换无刷新
3. **暗色模式** — 自动跟随系统，也支持手动切换
4. **功能刚好** — 搜索、目录、字数统计、评论系统——该有的都有，没有多余的
5. **文档质量** — 写得清楚，有示例，有说明

**一个判断主题好坏的简单方法：看它的文档。** 文档写得好的项目，代码质量通常也不会差。

## 配置的取舍

### 写作体验 vs 功能丰富

我花了很多时间在"要不要启用某个功能"上。最后的原则是：**如果一个功能不能让写作体验变好，就不需要。**

- 字数统计 ✅ — 让读者知道文章长度
- 文章目录 ✅ — 长文章必备
- 评论系统 ❌ — 先不加，等有读者了再说
- 文章推荐 ❌ — 内容太少，推荐没意义
- 音乐播放器 ❌ — 博客不是音乐网站

### 个性化 vs 维护成本

自定义 CSS 和 JS 是好东西，但每多一行自定义代码，就多一行需要维护的代码。我的原则是：

- **改颜色和字体** — 值得，因为这是品牌识别
- **改布局和组件** — 不值得，除非主题自带的功能真的不够
- **添加交互效果** — 少量可以，多了就是负担

## 部署方案

我选择了宝塔面板 + 阿里云的组合，原因：

1. **完全控制** — 不受第三方平台限制
2. **成本可控** — 一台轻量服务器够用
3. **学习价值** — 部署过程本身就是学习

其他方案的取舍：

- **GitHub Pages** — 免费，但国内访问慢
- **Vercel** — 方便，但怕被墙
- **Cloudflare Pages** — 好用，但自定义域名配置稍复杂

## 搭建过程中的坑

### 坑 1：主题配置文件的位置

Redefine 的配置文件可以放在 `_config.redefine.yml`（站点根目录）或 `source/_data/redefine.yml`。我一开始放在了错误的位置，导致配置不生效。

**教训：先读文档，再动手。**

### 坑 2：图片路径

Hexo 的 `post_asset_folder` 选项会影响图片引用方式。我选了 `true`，这样每篇文章的图片可以放在同名文件夹里，管理更清晰。

### 坑 3：中文标签的 URL 编码

中文标签生成的 URL 会变成 `%E6%A0%87%E7%AD%BE` 这种，不好看。解决方案是在 `_config.yml` 里配置 `pretty_urls`。

## 写在最后

搭博客这件事，技术含量其实不高。真正难的是：**搭完之后坚持写。**

我给自己定了一个目标：每周至少写一篇文章。不求多长多深，但要持续输出。因为博客的价值不在于技术有多炫酷，而在于内容的积累。

如果你也在考虑搭博客，我的建议是：**别想太多，先搭起来。** 哪怕只有一篇文章，那也是你的第一块砖。以后的每一行文字，都是在上面添砖加瓦。
