## 起因

ChatGPT 很强，但它有一个致命问题：**它不知道你的文档里写了什么。**

你上传一份内部技术文档，问它"我们的认证流程是怎样的"，它要么瞎编，要么说"我没有访问你的文档的能力"。

RAG（Retrieval-Augmented Generation）就是为了解决这个问题：先从文档中检索相关内容，再把检索结果交给大模型生成回答。这样大模型的回答就是"有据可依"的。

我想自己做一遍这个流程，于是有了 DocMind。

## 系统架构

DocMind 的核心是一个 RAG Pipeline：

```
用户提问
   │
   ▼
┌──────────────┐
│  意图理解     │  ← 判断是否需要检索
└──────┬───────┘
       │
       ▼
┌──────────────┐
│  混合检索     │  ← BM25 关键词 + 向量语义
└──────┬───────┘
       │
       ▼
┌──────────────┐
│  重排序       │  ← BGE Reranker 精排
└──────┬───────┘
       │
       ▼
┌──────────────┐
│  生成回答     │  ← 大模型 + 检索上下文
└──────┬───────┘
       │
       ▼
   引用溯源
```

### 为什么用混合检索

纯向量检索有个问题：它擅长语义匹配，但不擅长精确匹配。比如你搜"ERR-4032 错误码"，向量检索可能返回一堆"错误处理"相关的内容，但就是找不到那个具体的错误码。

BM25 关键词检索正好相反：精确匹配强，语义理解弱。

混合检索把两者结合起来，用 RRF（Reciprocal Rank Fusion）融合排序，取长补短。

### 文档处理 Pipeline

上传文档后的处理流程：

1. **解析** — PDF/Word/PPT/TXT，支持扫描件（PaddleOCR）
2. **分块** — 按语义分块，不是简单按字数切
3. **向量化** — Embedding 模型生成向量
4. **索引** — 存入 Elasticsearch（全文）+ 向量数据库

## 技术栈选择

### 后端：FastAPI

选 FastAPI 而不是 Flask/Django，理由：

- **异步原生** — RAG Pipeline 涉及大量 I/O（调 Embedding、调大模型、查数据库），异步能显著提升吞吐量
- **自动文档** — Swagger UI 自动生成 API 文档，前后端联调省很多事
- **类型安全** — Pydantic 模型做参数校验，减少运行时错误

### 前端：Vue 3 + Naive UI

Vue 3 的 Composition API 写起来很舒服，Naive UI 的组件质量高，开箱即用。

### 基础设施

- **MySQL** — 用户、权限、元数据
- **Elasticsearch** — 全文检索 + 向量检索
- **Redis** — 缓存热数据
- **MinIO** — 文件存储（S3 兼容）
- **Kafka** — 异步任务队列（文档解析是耗时操作）

## 踩过的坑

### 1. 分块策略直接影响检索质量

一开始用固定字数分块（500 字一块），效果很差。一个问题的答案经常被切到两个块里，导致检索不到完整信息。

后来改成**语义分块**：按段落、标题等自然边界切分，每个块保留上下文重叠。检索准确率提升明显。

### 2. Elasticsearch 向量检索的内存问题

Elasticsearch 的向量检索（dense_vector）在数据量大时非常吃内存。10 万条文档的向量索引占了 8GB 内存。

解决方案：降维（从 1536 维降到 768 维）+ 量化（float32 → int8），内存占用降到 2GB，精度损失在可接受范围内。

### 3. 大模型的"幻觉"

即使检索到了正确的内容，大模型仍然可能"创造性地"回答。比如检索到"认证流程使用 JWT"，大模型可能回答成"认证流程使用 OAuth2.0"。

解决方案：在 prompt 中明确要求"只基于提供的上下文回答，如果上下文没有相关信息，请说'我没有找到相关信息'"。加上引用溯源，让用户自己验证。

## 成果

- 支持 PDF/Word/PPT/TXT 文档上传和解析
- 混合检索（BM25 + 向量 + RRF 融合 + BGE Reranker）
- SSE 流式对话，实时展示生成过程
- 引用溯源，每个回答都能追溯到原文
- Docker Compose 一键部署

项目地址：[github.com/sijie-Z/DocMind-RAG](https://github.com/sijie-Z/DocMind-RAG)

## 总结

做 DocMind 最大的体会是：**RAG 的核心不是大模型，是检索。** 检索质量直接决定回答质量。花 80% 的时间在检索优化上（分块策略、混合检索、重排序），比花 80% 的时间在 prompt 工程上效果好得多。

这也是为什么我把项目命名为"DocMind"——让文档拥有"智能"，而不是让大模型拥有"文档"。
